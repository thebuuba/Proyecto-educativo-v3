# Flujo de autenticación en frontend

## AuthProvider

`AuthProvider` restaura la sesión llamando a `/api/v1/auth/bootstrap`. El navegador envía automáticamente la cookie `aulabase_session`; JavaScript no puede leerla porque es `HttpOnly`.

```text
App inicia
  └─ GET /auth/bootstrap con credentials: include
      ├─ 200: carga usuario, roles, permisos y onboarding
      └─ 401: intenta restaurar Supabase; si no existe sesión, queda no autenticado
```

El login por correo y OAuth obtiene primero una sesión de Supabase. El access token de Supabase se usa únicamente para intercambiarlo en `/auth/session`; el backend valida la identidad y fija su JWT propio en cookie.

## Seguridad

- No se guarda JWT en `localStorage`.
- `apiClient` usa rutas relativas `/api/v1` y `credentials: include`.
- La cookie es `HttpOnly`, `SameSite=Lax` y `Secure` en producción.
- `SUPABASE_SERVICE_ROLE_KEY` nunca existe en el frontend.
- La anon key de Supabase es pública y las políticas RLS/Storage definen el acceso real.

`RequireAuth` protege las rutas según el estado, los roles y los permisos cargados por `AuthProvider`.
